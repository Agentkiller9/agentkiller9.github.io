Detecting an active cloud server breach

Threat hunting · Incident response · SecOps Internship · 2025

DFIR Threat Hunting Incident Response Cloud Linux Forensics

# TL;DR

During day-to-day monitoring on the SecOps internship rotation, I identified anomalous activity on a university-managed cloud server that turned out to be an active, in-progress compromise. I escalated, helped triage and contain the incident, and walked through the forensic timeline with the team afterwards.

# Context

  • Role: Security Operations Intern, APU SecOps Functional Unit.
  • Asset class: University-owned Linux cloud server.
  • Detection surface: Routine review of system, authentication and process telemetry as part of ongoing threat-hunting rotations.

Hostnames, IPs, indicators of compromise and exploit details are redacted. This page covers the methodology only.

# Detection

The initial signal was anomalous activity that didn't match the host's normal behavioural baseline. Specifically:

  • Unexpected outbound connections from a service account that had no business making them.
  • Process activity originating from a directory typically used only for short-lived staging.
  • Authentication-log patterns inconsistent with the documented administrator workflow.

Cross-referencing those indicators against MITRE ATT&CK techniques and historical baseline made it clear this wasn't a misconfiguration — something was actively present.

# Triage

1. Confirm, don't tip off

Validated the indicators using read-only telemetry before touching the host directly — the goal was confirmation without altering evidence or alerting the attacker.

2. Escalate immediately

Once confidence crossed the threshold, escalated to the system administrator and SecOps lead with a written summary of findings, timestamps and the next-step options.

3. Scope the blast radius

Helped enumerate the services and accounts the compromised host could reach, the credentials it held, and the trust relationships that might let the attacker pivot.

4. Contain

Supported the containment plan — network isolation, credential rotation for in-scope accounts, and snapshotting state for later forensic review.

5. Reconstruct the timeline

Used Windows / Linux event-log analysis (the same workflow I later automated in Chainsaw Automator) to walk back the attacker's actions and identify the initial access vector.

# Outcome

  • The compromise was contained without further lateral movement out of scope.
  • Root cause and IoCs were documented and fed into the team's detection rules.
  • The experience directly motivated my final-year project Jinsoku — a self-hosted DFIR workbench that wraps Hayabusa and Takajo behind a workflow built around exactly this kind of incident.

# Lessons learned

  • Baselines beat alerts. The detection didn't come from a tuned SIEM rule — it came from knowing what "normal" looked like for this host.
  • Quiet confirmation first. If you act before you're sure, you destroy evidence and burn detection. If you wait too long, the attacker moves. The discipline is in the middle.
  • Tooling matters at 3am. Triaging this manually was slow. Jinsoku exists because the next analyst shouldn't have to re-do every step from a blank terminal.
